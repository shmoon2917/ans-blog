# 웹 크롤러가 무엇인가?

웹 사이트를 검색 결과에 표시하기 위해 Google(Bing, Yandex, Baidu, Naver, Yahoo 또는 DuckDuckGo와 같은 다른 검색 엔진들도)은 웹 크롤러를 사용하여 웹 사이트를 탐색합니다.

서로 다른 검색 엔진들은 각 나라별로 상이한 점유율을 가집니다.

이 가이드에서는 대부분의 국가들에서 가장 큰 검색 엔진인 구글을 다뤄볼 예정입니다.

Ranking과 Rendering의 경우, 검색 엔진 간의 일부 차이가 있을 수 있지만, Crawling과 Indexing의 경우에는 대부분의 검색 엔진들이 거의 비슷한 방식으로 동작합니다.

웹 크롤러는 유저를 모방하고 웹 사이트에 있는 링크를 탐색하여 페이지를 인덱싱하는 봇의 한 종류입니다. 웹 크롤러는 커스텀 user-agent를 사용하여 자신을 식별합니다. 구글은 여러 개의 웹 크롤러를 가지고 있지만, 자주 사용되는 것은 **구글봇 데스크톱**과 **구글봇 스마트폰**입니다.

# 구글봇은 어떻게 동작하는가?

![The journey of GoogleBot {715x491}](/assets/articles/the-journey-of-GoogleBot.png)

개요는 다음과 같습니다.

1. **Find URLs**: Google은 Google 검색 콘솔, 웹 사이트 간 링크 또는 [XML sitemaps](https://developers.google.com/search/docs/advanced/sitemaps/overview)을 비롯한 여러 위치에서 URL을 얻습니다.
2. **Add to Crawl Queue**: 이 URL들은 구글봇이 처리할 크롤링 큐에 추가됩니다. 큐의 URL들은 보통 몇 초 동안 지속되는데, 특히 페이지를 렌더링, 인덱싱해야 하는 경우, 또는 리프레싱(URL이 이미 인덱싱된 상태)해야 하는 경우 며칠까지 지속될 수 있습니다. 페이지들은 이후 Render Queue에 추가됩니다.
3. **HTTP Request**: 헤더를 얻기 위해 HTTP 요청을 하고, 리턴된 status code에 따라 행동합니다.

   - `200` - HTML 파일을 크롤링 및 파싱합니다.
   - `30X` - 리다이렉트를 준수합니다.
   - `40X` - 에러를 기록하고 HTML을 로드하지 않습니다.
   - `50X` - status code가 변경된다면 아마 다시 방문할 겁니다.

4. **Render Queue**: 검색 시스템의 다양한 서비스와 컴포넌트들이 HTML을 처리하고 컨텐츠를 파싱합니다. 페이지에 JavaScript 클라이언트 기반 콘텐츠가 있는 경우 URL을 렌더 큐에 추가할 수 있습니다. 렌더링 큐는 JavaScript를 렌더링하기 위해 더 많은 리소스를 사용해야 하기 때문에 구글에게 더 많은 비용이
   듭니다. 따라서 렌더링되는 URL은 인터넷 상의 전체 페이지에서 차지하는 비율이 낮습니다. 일부 다른 검색 엔진은 Google과 같은 렌더링 용량을 가지고 있지 않을 수 있으며, 여기서 Next.js가 렌더링 전략을 지원할 수 있습니다.
5. **Ready to be indexed**: 모든 기준들이 충족되었다면, 페이지는 인덱싱되고 검색 결과에 표시될 자격을 얻습니다.
